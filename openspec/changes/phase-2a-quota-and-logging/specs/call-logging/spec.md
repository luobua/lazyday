## ADDED Requirements

### Requirement: Call log persistence

The system SHALL persist a record for every request that traverses `RateLimitWebFilter` and `CallLogWebFilter`. Records SHALL be written to the partitioned `t_call_log` table with month-based range partitions on `request_time`.

#### Scenario: Successful request logged

- **WHEN** an authenticated tenant calls any matched API endpoint and receives any HTTP response
- **THEN** within 5 seconds a row exists in `t_call_log` with the matching `tenant_id`, `app_key`, `path`, `method`, `status_code`, `latency_ms`, `client_ip`, and `request_time`

#### Scenario: app_key resolution by traffic flow

- **WHEN** an Open API request carries `X-App-Key: ak_xxx` (HMAC flow, Phase 2b)
- **THEN** the row's `app_key` SHALL be the literal AppKey value `ak_xxx`
- **WHEN** a Portal API request is authenticated only by JWT (no `X-App-Key`)
- **THEN** the row's `app_key` SHALL be the sentinel `__PORTAL__`
- **WHEN** an Admin API request is authenticated only by JWT
- **THEN** the row's `app_key` SHALL be the sentinel `__ADMIN__`

The sentinels `__PORTAL__` / `__ADMIN__` SHALL never collide with real AppKeys because Phase 1 mandates the `ak_` prefix for all generated AppKeys.

#### Scenario: Asynchronous write does not block response

- **WHEN** the call log write encounters a transient failure
- **THEN** the response to the client SHALL NOT be delayed or fail; the failure is logged at WARN level with metric `lazyday.calllog.write.failed` incremented

#### Scenario: Backpressure shedding under sustained burst

- **WHEN** the in-process backpressure buffer (`Sinks.Many.onBackpressureBuffer`, default capacity 10000) is full and another log entry is offered
- **THEN** the entry SHALL be dropped, the response to the originating request SHALL NOT be affected, and metric `lazyday.calllog.write.shed` SHALL be incremented at WARN level
- **AND** as the consumer drains the buffer, subsequent entries SHALL be accepted again

#### Scenario: Buffer capacity is configurable

- **WHEN** `service.callLogBufferSize` is set in application configuration
- **THEN** the in-process backpressure buffer SHALL be sized accordingly at application startup

#### Scenario: Snowflake ID generation

- **WHEN** a call log row is written
- **THEN** its `id` SHALL be generated by the application's Snowflake ID generator (NOT a database identity column)

#### Scenario: error_msg captured on error responses

- **WHEN** the downstream filter chain emits `onError` (uncaught exception) OR the response status code is ≥ 500
- **THEN** the row's `error_msg` SHALL contain the exception class name + a colon + the exception message, truncated to 500 characters; for status ≥ 500 without an exception the field contains the response status reason phrase
- **WHEN** the response status code is < 500 and no exception was thrown
- **THEN** the row's `error_msg` SHALL be NULL

### Requirement: Monthly partitioning

The system SHALL maintain monthly partitions of `t_call_log` named `t_call_log_YYYY_MM`. The system SHALL ensure that at least the current month and the next month always have an existing partition.

#### Scenario: Initial partitions created by migration

- **WHEN** Flyway migration V3 executes
- **THEN** partitions for the current month and the next month exist

#### Scenario: Migration fails fast on PostgreSQL < 11

- **WHEN** Flyway migration V3 is applied against a PostgreSQL instance with `server_version_num < 110000`
- **THEN** the migration SHALL raise an exception immediately and abort before creating any tables, with an error message naming the required version

#### Scenario: Monthly scheduler creates upcoming partition

- **WHEN** the `PartitionScheduler` runs on the last day of any month
- **THEN** the partition for the month after next is created with naming `t_call_log_YYYY_MM`

#### Scenario: Application startup ensures upcoming partitions

- **WHEN** the Backend application starts
- **THEN** it ensures the next two months of partitions exist, creating any missing partition

#### Scenario: Write to nonexistent partition

- **WHEN** the system attempts to write a call log row whose `request_time` falls outside any existing partition
- **THEN** the system raises error_code `PARTITION_MISSING`, logs a critical alert, and does not crash the request being served

### Requirement: Call log query and aggregation

The system SHALL provide tenant-scoped queries for paginated call logs and aggregated statistics.

#### Scenario: Paginated log query

- **WHEN** an authenticated TENANT_ADMIN calls `GET /api/portal/v1/logs?startTime=...&endTime=...&page=1&size=20`
- **THEN** the system returns up to 20 log entries scoped to the user's tenant_id, ordered by `request_time DESC`, with pagination metadata

#### Scenario: Required time range parameter

- **WHEN** a tenant calls `GET /api/portal/v1/logs` without `startTime` and `endTime`
- **THEN** the system returns HTTP 400 with error_code `INVALID_PARAMETER` and message indicating the time range is required

#### Scenario: Filter by status code

- **WHEN** a tenant adds `statusCode=429` to the log query
- **THEN** the response only includes records with `status_code=429`

#### Scenario: Hourly aggregation

- **WHEN** a tenant calls `GET /api/portal/v1/logs/stats?startTime=...&endTime=...&granularity=hour`
- **THEN** the system returns one bucket per hour in the range, each with `total_count`, `success_count` (status_code 2xx/3xx), and `error_count` (status_code 4xx/5xx)

#### Scenario: Daily aggregation

- **WHEN** a tenant calls `GET /api/portal/v1/logs/stats?startTime=...&endTime=...&granularity=day`
- **THEN** the system returns one bucket per day in the range with the same fields as hourly aggregation

#### Scenario: Tenant data isolation

- **WHEN** any log query API executes
- **THEN** the WHERE clause SHALL include `tenant_id = <jwt-context-tenant-id>` regardless of query parameters

### Requirement: Call log data model

The `t_call_log` table SHALL contain: id (BIGINT PK, snowflake), tenant_id (BIGINT NOT NULL), app_key (VARCHAR 64 NOT NULL — real `ak_*` value, or sentinel `__PORTAL__` / `__ADMIN__` for JWT flows), path (VARCHAR 500), method (VARCHAR 10), status_code (SMALLINT), latency_ms (INT), client_ip (VARCHAR 50), error_msg (TEXT NULL), request_time (TIMESTAMPTZ NOT NULL DEFAULT NOW()). The table SHALL be partitioned BY RANGE (request_time). Indexes `idx_call_log_tenant_time(tenant_id, request_time)` and `idx_call_log_app_key(app_key)` SHALL exist.

#### Scenario: Table structure after migration

- **WHEN** Flyway migration V3 executes
- **THEN** `t_call_log` exists with all specified columns, partitioning, and indexes
